## 一、Seata 分布式事务模式

### 速记口诀

```
AT自动TCC手，SAGA长来XA慢
```

### 4 种模式对比

| 模式 | 全称 | 特点 | 侵入性 | 性能 | 适用场景 |
|------|------|------|--------|------|----------|
| AT | Automatic Transaction | 自动回滚，用 undo_log | 无 | 高 | 大多数场景 |
| TCC | Try-Confirm-Cancel | 手写三个方法 | 高 | 中 | 强一致要求 |
| SAGA | SAGA | 长事务，每步有补偿 | 中 | 中 | 长流程业务 |
| XA | XA Protocol | 数据库两阶段提交 | 无 | 低 | 传统数据库 |

### AT 模式原理（重点）

1. 执行前：记录数据快照到 undo_log
2. 执行业务 SQL
3. 成功 → 删除 undo_log
4. 失败 → 用 undo_log 自动生成反向 SQL 回滚

```
INSERT → DELETE
UPDATE → UPDATE（改回原值）
DELETE → INSERT（插回原数据）
```

### TCC 模式（了解）

需要手写三个方法：

```java
public void tryDeduct() { }    // Try：预留资源
public void confirmDeduct() { } // Confirm：真正执行
public void cancelDeduct() { }  // Cancel：回滚释放
```

### 面试回答模板

> 项目用的是 AT 模式，因为它无代码侵入，通过 undo_log 自动回滚。TCC 需要手写 Try/Confirm/Cancel 三段代码，适合强一致性场景。SAGA 适合长流程业务。XA 是数据库标准协议，但性能差，很少用。

## 二、Spring 事务传播行为

### 速记口诀

```
REQUIRED一起，NEW独立，NESTED嵌套
```

### 3 种常用传播行为

#### 1. REQUIRED（默认）

- 有事务就加入，没有就新建
- 同一个事务，同生共死
- 使用场景：大多数业务

```java
@Transactional  // 默认就是 REQUIRED
public void createOrder() {
    // 多个操作一起成功或失败
}
```

#### 2. REQUIRES_NEW

- 总是新建独立事务
- 和外部事务互不影响
- 使用场景：操作日志、发送通知

```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void saveLog() {
    // 无论主业务成功失败，日志都保留
}
```

#### 3. NESTED

- 嵌套事务，基于 Savepoint
- 内部失败只回滚内部
- 外部失败全部回滚
- 使用场景：批量处理

```java
@Transactional(propagation = Propagation.NESTED)
public void processItem() {
    // 单个失败不影响整体
}
```

### REQUIRES_NEW 和 NESTED 区别（重点）

| 场景 | REQUIRES_NEW | NESTED |
|------|--------------|--------|
| 内部失败 | 只回滚内部 | 只回滚内部 |
| 外部失败 | **内部不回滚** | **内部也回滚** |
| 关系 | 完全独立 | 父子关系 |

### 面试回答模板

> REQUIRED 是默认行为，有事务就加入，没有就新建，同事务同命运。REQUIRES_NEW 用于日志等场景，总是新建独立事务，外部失败不影响它。NESTED 是嵌套事务，内部失败可单独回滚，但外部失败会一起回滚。

## 三、高频面试题

### Q1：AT 模式如何实现回滚？

> AT 模式通过 undo_log 表记录数据的前后快照。事务失败时，Seata 会根据快照自动生成反向 SQL 进行回滚，INSERT 变 DELETE，UPDATE 变改回原值的 UPDATE。

### Q2：TCC 和 AT 有什么区别？

> AT 模式无代码侵入，自动回滚，适合大多数场景。TCC 需要手写 Try、Confirm、Cancel 三个方法，代码侵入性强，但可以做到更强的一致性控制。

### Q3：REQUIRES_NEW 和 NESTED 有什么区别？

> REQUIRES_NEW 创建完全独立的事务，外部事务失败不影响它。NESTED 是嵌套事务，基于数据库 Savepoint，外部失败时内部也会回滚。

### Q4：什么场景用 REQUIRES_NEW？

> 操作日志、发送通知、消息记录等场景。这些操作无论主业务成功失败都需要保留，所以用独立事务。

### Q5：什么场景用 NESTED？

> 批量处理场景，比如批量创建订单明细。单个失败只回滚该条，不影响其他明细和主订单。
